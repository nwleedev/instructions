# 운영 스택 하네스 VALIDATION

## 목표

운영 스택 문서가 “좋아 보이는 원칙”에 머물지 않고, 실제로 현재 저장소와 향후 패키지화 결정을 지지하는지 확인한다.

## 완료 체크리스트

### 1. 계층 분리

- 머신/사용자/저장소/세션/팀/배포 계층이 구분돼 있다.
- 개인 기본값과 프로젝트 오버라이드의 경계가 드러난다.
- 코어/어댑터/evidence 경계가 문서에 반영돼 있다.

### 2. 운영 루프

- 작업 시작과 종료의 세션 절차가 설명된다.
- 반복 작업을 스킬/템플릿으로 승격하는 규칙이 있다.
- 검증/리뷰 루프가 별도 단계로 드러난다.

### 3. 권한과 통제

- 승인 정책, 샌드박스, MCP, 훅/스킬이 운영 표면으로 정리돼 있다.
- 공식 공개 표면과 내부 추정 표면이 섞이지 않는다.
- 보안 기본값이 문서에 적혀 있다.

### 4. 배포 구조

- 현재 복사형 모델의 장단점이 명시돼 있다.
- 장기 패키지형 모델의 장단점이 명시돼 있다.
- 패키지 구조, 업그레이드 흐름, override 보존 전략이 문서화돼 있다.

### 5. 유지 판단

- `harness-engine`, `instructions/*.md`, `instructions/templates/*.md`를 유지할지 판단이 있다.
- `sync-harness.sh`를 즉시 폐기하지 않는 이유가 있다.
- 후속 구현 티켓으로 분리해야 할 항목이 명시돼 있다.

### 6. 세션 산출물

- `SYSTEM-SETUP`, `WORKFLOW-PLAYBOOK`, `HARNESS-DECISION`, `DISTRIBUTION-RFC`, `PACKAGE-ARCHITECTURE`, `MIGRATION-PLAN`이 존재한다.
- validation artifact가 세션 경로에 있다.
- 다음 작업자가 RFC만 읽고도 구현 티켓을 열 수 있다.

## 종합 판정 기준

- **통과**
  - 운영 계층, 권한 모델, 배포 구조, 유지 판단이 모두 닫힘
  - 세션 RFC 묶음과 validation artifact 존재
- **조건부 통과**
  - 장기 방향은 닫혔지만 구체 구현 티켓이 아직 필요한 경우
  - 미구현 항목을 후속 작업으로 명시했을 때만 허용
- **실패**
  - 운영 계층 구분 부재
  - 배포 구조 비교 부재
  - 유지/폐기/전환 판단 부재

## 권장 검증 순서

1. 공식 문서 기반 운영 표면 점검
2. 현재 저장소 자산 목록 점검
3. 유지/개편 판단 점검
4. distribution RFC와 package architecture의 일관성 점검
5. migration plan의 단계별 실행 가능성 점검
