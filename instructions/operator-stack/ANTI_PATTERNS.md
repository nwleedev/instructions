# 운영 스택 하네스 ANTI_PATTERNS

## 목표 혼동

| 케이스 | Anti | Good |
|---|---|---|
| 내부 기능 추격 | Anthropic 내부 전용 기능을 재현하려고 운영 스택을 설계 | 외부 사용자가 구축 가능한 공개 표면 위주로 설계 |
| 배포-규칙 혼동 | 배포 UX가 불편하니 하네스 규칙 전체를 버림 | 규칙층과 배포 메커니즘을 분리해 판단 |
| 단일 도구 종속 | Claude Code 또는 Codex 하나에만 맞춘 운영 구조 | 공통 운영 계층과 벤더별 adapter를 분리 |

## 설정 계층

| 케이스 | Anti | Good |
|---|---|---|
| 일회성 플래그 남용 | 모든 설정을 커맨드라인 플래그로만 운용 | 사용자 기본값과 프로젝트 오버라이드를 구분 |
| 개인/프로젝트 혼합 | 개인 비밀 설정과 저장소 규칙을 한 파일에 섞음 | user config와 repo config를 분리 |
| 권한 불명확 | 승인 정책과 샌드박스 수준이 문서에 없음 | 권한 모델을 운영 스택의 필수 축으로 기록 |

## 작업 흐름

| 케이스 | Anti | Good |
|---|---|---|
| 세션 부재 | 장기 작업인데 세션 기록 없이 진행 | 세션 문서를 작업 게이트로 사용 |
| 리뷰 루프 없음 | 구현 후 끝내고 검증/리뷰 단계를 생략 | 테스트/리뷰/회고를 명시적 단계로 유지 |
| 반복 작업 수동화 | 자주 하는 작업을 매번 프롬프트로만 설명 | 스킬, 템플릿, contract packet으로 승격 |

## 배포 모델

| 케이스 | Anti | Good |
|---|---|---|
| 무조건 복사 유지 | `sync-harness.sh`만 계속 늘려 감 | 버전 관리와 업그레이드 경로를 갖춘 패키지형 모델 검토 |
| 무근거 패키지화 | 현재 자산 구조를 정리하지 않고 바로 npm 패키지 구현 | 코어/어댑터/evidence 경계를 먼저 문서화 |
| 전역 덮어쓰기 | 프로젝트 로컬 override를 보존하지 않음 | 마커 병합, 선택 설치, 비관리 자산 보존 전략 유지 |

## 문서화

| 케이스 | Anti | Good |
|---|---|---|
| 추상적 플레이북 | “좋은 운영 환경을 만든다” 수준에서 끝남 | 시스템 계층, 명령 흐름, 검증 포인트를 구체화 |
| 출처 부재 | 운영 권장안을 경험담처럼만 서술 | 공식 문서와 현재 저장소 구조를 함께 인용 |
| 마이그레이션 공백 | 장기 방향만 말하고 단계별 전환안이 없음 | RFC와 migration plan을 함께 유지 |
