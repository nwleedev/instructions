# `instructions/FAILURE.md`

작업 중 오류, 테스트 실패, 빌드 실패, 타입 오류, 런타임 예외가 발생하면 먼저 해당 문제의 **재현 조건과 직접 원인**을 식별한다.
직접 원인을 해결하지 않은 상태에서 우회 구현 금지, 기능 비활성화 금지, 요구사항 해석 변경 금지, 목표 축소 금지

문제를 해결하지 못했다고 해서 문제를 숨기거나 회피하는 방식으로 작업을 완료 처리하는 대신 다른 방법을 조사하고 대응한다.

파일 기록 위치

- 진행 영향: PROGRESS.md
- 중요한 판단: DECISIONS.md
- 개별 작업 상태: TICKETS.md

### 실패 대응 원칙

- 문제 발생 시 반드시 수행

  - 실패 증상 기록
  - 재현 절차 기록
  - 직접 원인 또는 가장 가능성이 높은 원인 후보 식별
  - 원인별 대응책 비교

- 대응책 비교 시 고려 항목

  - 애플리케이션 목표 유지 여부
  - 테스트 영향
  - 구현 범위 영향
  - 되돌리기 가능성
  - 임시 대응 여부

- 해당 사항의 경우 더 나은 방법을 조사해서 작업을 진행하세요.

  - 에러를 숨기는 처리만 추가
  - 테스트를 비활성화하여 통과 처리
  - 기능 제거로 실패를 사라지게 만드는 방식
  - 문제를 기록하지 않고 티켓을 완료 처리

- 예외
  - 외부 서비스 장애
  - 보안상 즉시 차단이 필요한 상황
  - 이 경우에도 임시 대응임을 명시하고 후속 티켓을 세팅.

---

### 목표 불변성과 제약 분리

애플리케이션 목표와 티켓 목표는 구현 과정에서 발견한 라이브러리, 프레임워크, 도구의 제약 때문에 임의로 변경하지 않습니다.

구현 수단의 한계는 제품 목표 변경 사유가 아니라 **구현 제약 정보**입니다.
라이브러리의 한계를 발견했다는 이유만으로 제품 목표나 acceptance criteria를 축소하거나 재정의하면 안 됩니다.

- 라이브러리 제약 발견 시 처리 순서

  - 애플리케이션 목표 재확인
  - 티켓 목표 재확인
  - 현재 구현 방식으로 목표 달성 가능 여부 검토
  - 대체 구현 방식 조사

- 대체 구현 방식 조사 범위

  - 라이브러리 설정 또는 사용 방식 변경
  - 다른 라이브러리 또는 도구
  - 자체 구현
  - 작업 분할 또는 선행 티켓 추가

- 목표 변경이 필요한 경우

  - 목표 변경을 직접 수행하지 않음
  - 사용자 검토 질문 생성
  - DECISIONS 기록 추가

- 금지
  - 라이브러리 한계를 제품 요구사항으로 간주
  - 라이브러리에 맞춰 acceptance criteria 재정의
  - 구현 편의를 위해 목표 축소

### 실패 분석 기록 규칙

문제가 발생한 경우 원인 분석과 대응 과정은 반드시 기록합니다.
문제를 해결했더라도 분석 기록을 남기지 않으면 동일 문제가 반복될 가능성이 높습니다.

- 문제 분석 기록 항목

  - 증상
  - 재현 절차
  - 직접 원인 또는 원인 후보
  - 고려한 대응책
  - 선택한 대응책
  - 선택 이유

- 대응책 비교 시 포함할 관점

  - 목표 유지 여부
  - 테스트 영향
  - 구현 복잡도
  - 되돌리기 가능성

- 기록 위치
  - PROGRESS
  - DECISIONS
  - 관련 티켓

## Blocked 상태 규칙

직접 원인을 확인했지만 현재 환경, 권한, 외부 시스템 제약 때문에 해결할 수 없는 경우 해당 작업을 **blocked 상태**로 전환합니다.

blocked 상태에서는 문제를 우회 구현으로 숨기지 않습니다.

- blocked 전환 조건

  - 문제 재현 가능
  - 직접 원인 또는 원인 후보 확인
  - 현재 세션에서 해결 불가

- blocked 기록 항목

  - 문제 증상
  - 직접 원인
  - 시도한 해결 방법
  - 해결에 필요한 외부 결정 또는 권한
  - 후속 티켓 여부

- 금지
  - blocked 사유를 숨기고 완료 처리
  - 임시 우회 구현만 남기고 원인 미기록

---

## 문제 회피 금지 규칙

문제를 해결하지 못했을 때 다음 행동은 허용되지 않습니다.

- 문제 원인을 해결하지 않고 구현 방식만 변경
- 기능을 제거하여 테스트 통과
- 요구사항을 축소하여 완료 처리
- 문제를 다른 티켓으로 넘기고 현재 티켓 완료 처리

문제를 해결할 수 없는 경우에는 반드시 **blocked 상태로 전환하거나 사용자 검토를 요청합니다.**
