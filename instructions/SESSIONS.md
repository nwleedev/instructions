# `instructions/SESSIONS.md`

**작업 맥락과 흐름을 오랜 기간 유지하기 위해 사용하는 세션에 대한 것입니다.**

## 세션에 대해서

- 사용자는 다음과 같은 방법로 Codex, Claude를 실행할 때 해당 세션 ID에 대해서 작업을 시작하거나 이어나갈 수 있음.

  - 명령어 예시
    - `codex resume <session_id>`
    - `claude --resume <session_id>`
  - 사용자는 `SESSION_ID=<session_id>` 환경변수를 같이 주입.
  - 세션 ID는 환경변수 SESSION_ID로 파악.
  - (`codex resume <session_id_1>` 또는 `claude --resume <session_id_1>`)과 `SESSION_ID=<session_id_2>`에서, session_id_1과 session_id_2는 서로 같다고 가정할 것.

- 인터넷 연결 요청에 대해서는 세션 ID를 확인하지 않음.(세션 ID를 확보하기 위한 작업)
- 만약 세션 아이디가 없다면 사용자에게 세션을 임시로 세팅할지를 질문

  - 세팅한다면 현재 작업에 대해서 임의로 세션 아이디를 세팅하고 진행
  - 세팅하지 않는다면 무시하고 진행.

- 사용자가 프롬프트를 Codex, Claude에 입력하여, Codex, Claude가 작업을 시작할 때, 그리고 끝낼 때마다 TICKETS.md(+ tickets 폴더), PROGRESS.md, DECISIONS.md(+ decisions 폴더), RESEARCH.md에 내용을 적절한 위치에 기록한다.
  - 작업을 시작할 때에도 세션 파일을 기록하는 이유는 작업 도중 컨텍스트 압축이 발생했을 때 작업 및 대화 맥락과 흐름을 유지하기 위함.
  - "만약 갑작스럽게 컨텍스트 압축(compaction)이 발생한다면, 우리가 작업 맥락과 대화 맥락을 유지하기 위해서는 어떤 파일에 어떤 내용을 기록해야 하는가"에 대해서 조사하고 세션 기록에 반영한다.
- `PLANS.md`는 작업 시작 가능 여부를 판정하는 기준 문서로 취급한다.
  - `PLANS.md`를 읽고 작업 완료에 필요한 정보가 비어 있거나 모호하면, 어떤 정보가 추가로 필요한지 사용자에게 제시한다.
  - 사용자가 정보를 전달하면 Codex, Claude는 기존 `PLANS.md`에 **추가하는 방식으로만** 반영한다.
  - `PLANS.md`의 기존 내용은 변경하거나 삭제하지 않는다.

**AGENTS.md 파일이 언급한 각 세션 파일에 대한 설명**

## PLANS.md

- `./store/<session_id>/PLANS.md` 파일
  - 이번 프로젝트에서 무엇을 하고, 무엇을 금지하는지를 정의하는 동시에, **작업 시작 가능 여부를 판정하는 실행 기준 문서**다.
  - 목표, 완료 기준뿐 아니라 **사용자 관점의 목적, 검증 방법, 범위 밖, 제약 조건, 작업 단계, 미정 사항 처리 방식**을 결정한다.
  - 우리가 진행할 작업의 범위는 해당 `PLANS.md` 파일에서 확인 가능하다.
  - 특정 작업이 해당 `PLANS.md`의 범위를 벗어나게 되면 작업 전 우선 사용자에게 검토 질문을 제공한다.
  - 작업 상태의 경우 `PLANS.md` 파일 대신 `TICKETS.md` 파일에 기록한다.
  - 장시간 작업에서도 기준 문서로 유지하며, Codex, Claude는 필요한 정보만 **append-only** 방식으로 추가할 수 있다.
  - 만약 작업을 진행하기 위해 기존 PLANS.md 파일에서 제거할 내용이 있을 경우 사용자에게 제거요청을 한다.

### `PLANS.md` 갱신 방식

- `PLANS.md`의 목표, 범위, 완료 기준, 금지 사항은 사용자 결정 문맥으로 취급한다.
- Codex, Claude는 `PLANS.md`를 읽고 부족하거나 충돌하는 항목을 식별할 수 있다.
- 작업 완료에 필요한 정보가 비어 있거나 모호하면, Codex, Claude는 `PLANS.md`에 어떤 정보가 추가로 필요한지 사용자에게 제시한다.
- 사용자가 선택지나 답변으로 정보를 전달하면, Codex, Claude는 그 내용을 기존 `PLANS.md`에 **추가하는 방식으로만** 반영한다.
- Codex, Claude는 `PLANS.md`의 기존 내용을 변경하거나 삭제하지 않는다.
- 새 정보가 기존 `PLANS.md`와 충돌해 기존 내용을 바꿔야 하는 경우에는 Codex, Claude는 임의로 수정하지 않고 사용자에게 검토를 요청한다.
- 작업 상태, 진행 로그, 조사 근거, 결정 사항은 각각 `TICKETS.md`, `PROGRESS.md`, `RESEARCH.md`, `DECISIONS.md`에 기록한다.

### 작업 시작 전 필수 항목

다음 항목이 비어 있거나 모호하면 바로 구현을 시작하지 않는다.

1. Purpose / Big Picture
   - 왜 이 작업을 하는가
   - 완료 후 사용자가 무엇을 할 수 있는가
   - 무엇을 보면 성공이라고 판단할 수 있는가
2. Scope
   - 이번 작업에 포함되는 것
   - 이번 작업에 포함되지 않는 것
   - 명시적으로 금지되는 것
3. Validation / Acceptance
   - 실행할 명령
   - 확인할 테스트, 빌드, 타입체크, 린트 범위
   - 수동 확인이 필요한 경우 관찰 포인트
4. Context and Orientation
   - 관련 파일, 경로, 모듈
   - 선행 제약과 참고 문서
   - 작업 이해에 필요한 용어 설명
5. Plan of Work
   - 수정 대상 파일
   - 단계 순서
   - 각 단계의 예상 산출물

### 사용자 관리 living document 필수 섹션

장시간 작업 또는 다단계 작업의 경우 다음 섹션을 유지한다. Codex, Claude는 이를 보강할 때도 기존 내용을 바꾸지 않고 필요한 정보만 뒤에 추가한다.

- `Progress`
- `Decision Log`
- `Surprises & Discoveries`
- `Outcomes & Retrospective`

### 정보 부족 시 질문 방식

- 필수 항목이 비어 있으면 Codex, Claude는 질문을 흩어 묻지 않고 한 번에 묶어서 제시한다.
- 질문은 `PLANS.md`에 추가 또는 명확화되어야 할 정보 기준으로 묶는다.
- 질문은 반드시 **선택지 형태**로 제시하며, 각 선택지마다 다음을 포함한다.
  - 언제 적합한지
  - 장점
  - 단점
  - 잠재적인 리스크
- 사용자가 선택지에 답한 뒤에는 Codex, Claude는 그 답을 기존 `PLANS.md`에 추가하는 방식으로 반영한다.
- 단순 구현 세부, 검색 중간 판단, 하위 단계 진행 순서 때문에 작업을 중단하지 않는다.
- 단순 구현 세부이거나 실질적인 선택지가 하나뿐인 경우에는 사용자 질문 없이 진행할 수 있다.
- 결과를 실질적으로 바꾸지 않는 하위 수준 미정 사항만 보수적이고 되돌리기 쉬운 방식으로 진행한다.
- 작업 시작 가능 판정에 필요한 핵심 정보가 비어 있으면 진행하지 않고, 필요한 항목을 사용자에게 제시한 뒤 응답받은 정보를 `PLANS.md`에 추가 반영한다.

### 질문이 필요한 대표 항목

- 완료 기준 강도
- 작업 범위 수준
- 외부 의존성 도입 여부
- 되돌리기 어려운 변경 허용 여부
- 산출물 형식
- 아키텍처 또는 테스트 전략 선택

### 기본 질문 세트

`PLANS.md`가 불완전할 때는 아래 질문 세트를 우선 사용한다. 질문 응답을 반영할 때는 기존 `PLANS.md`에 필요한 정보만 추가한다.

1. 완료 기준 강도 선택
   - 최소 검증: 구현 + 관련 테스트
   - 일반 검증: 구현 + lint + typecheck + 관련 테스트
   - 엄격 검증: 일반 검증 + build + 브라우저/E2E 또는 수동 검증
2. 작업 범위 선택
   - 기능 단위
   - 기능 + 구조 보정
   - 구조 개선 우선
3. 미정 사항 처리 방식 선택
   - 보수적 기본값 자동 선택
   - 중요한 항목만 질문
   - 모든 미정 항목 질문
4. 문서 강도 선택
   - 경량 템플릿
   - 표준 템플릿
   - 엄격 템플릿

### 작업 시작 가능 판정

다음 조건을 모두 만족할 때만 구현을 시작한다.

- 목표와 사용자 가시적 결과가 명확하다.
- 완료 기준이 검증 가능하다.
- 범위 밖 항목이 정의되어 있다.
- 관련 파일과 맥락이 적혀 있다.
- 남은 미정 항목이 결과를 실질적으로 바꾸지 않거나, 사용자 응답이 `PLANS.md`에 추가 반영되어 확정되었다.

## UPDATES.md

- `./store/<session_id>/UPDATES.md` 파일
  - `AGENTS.md`와 지침, 스킬을 읽을 때 수정 시각과 마지막 읽은 시각을 비교한다.
  - stale이면 다시 읽고 마지막 읽은 시각을 갱신한다.

### stale check

1. 대상 파일이나 폴더의 수정 시각을 확인한다.
2. `UPDATES.md`의 마지막 읽은 시각과 비교한다.
3. 마지막 읽은 시각이 더 이전이면 stale로 보고 다시 읽는다.
4. 다시 읽은 뒤 `UPDATES.md`의 마지막 읽은 시각을 갱신한다.

## Original Goal Check

세션 시작, 컨텍스트 압축 후 재개, 티켓 완료 직후에는 아래 3개를 반드시 갱신한다.

- Original User Goal:
- Current Best Next Ticket:
- Why This Advances The Original Goal:

규칙

- 위 3개가 비어 있으면 다음 티켓을 시작하지 않는다.
- 현재 티켓이 원래 목표를 직접 진전시키지 않으면, 보호 테스트/선행 정리/리스크 제거 중 무엇인지 명시한다.
- 원래 목표와 직접 연결되는 티켓이 남아 있는데 간접 티켓을 먼저 수행하려면, 이유를 `DECISIONS.md`에 남긴다.

## TICKETS.md

- `./store/<session_id>/TICKETS.md`
- 현재 무엇을 해야 하고, 무엇이 진행 중이며, 무엇을 완료했는지 판단하는 기준 문서이다.
- 각 항목에 대해서 상세 정보는 `./store/<session_id>/tickets` 폴더 내 중첩 폴더 없이 파일로 관리한다.
  - 파일 규칙: `TICKETS-####-{제목}.md`
    - 파일 내 날짜를 분 단위까지 포함하여 명시한다.
- 규칙
  - 작업 시작 전에 반드시 티켓으로 정의한다.
  - 다음과 같은 상황에서 즉시 업데이트한다.
    - 작업 시작할 때: 체크박스를 생성하거나 상태를 진행 중으로 표시
    - 작업이 막힌 경우: 별도의 아이콘으로 명시하고 사유를 한 줄로 기록한다.
    - 작업 완료: 체크박스 완료 처리
  - 상태 판단은 항상 TICKETS.md 파일 기준이다.
  - 작업의 상세 일지는 TICKETS 대신 PROGRESS.md 파일에 작성한다.
  - 의사결정의 근거 및 논의를 TICKETS에 적는 대신 DECISIONS.md 파일에 자세하게 기록한다.
- `TICKETS.md` 상단에는 항상 다음 섹션을 유지한다.
  - Original Goal
  - Current Best Next Ticket
  - Deferred But Important
- 티켓 완료 후에는 다음 티켓 후보가 Original Goal을 어떻게 진전시키는지 한 줄로 기록한다.

## 세션 기록의 최소 재개 정보

- 컨텍스트 압축, 세션 재시작, 장시간 작업 이후에도 바로 이어서 진행할 수 있도록 각 세션 기록에는 최소한 다음 정보가 남아 있어야 한다.
  - 현재 목표와 완료 조건
  - 지금까지 확인한 사실과 근거 문서
  - 이미 끝난 단계와 아직 남은 단계
  - 현재 막힌 지점이 있다면 그 직접 원인
  - 다음에 즉시 실행할 명령, 파일, 검증 절차
- 위 정보가 세션 파일에 기록되어 있다면, Codex, Claude는 사용자에게 같은 배경 설명을 다시 요구하지 않고 이어서 진행한다.
- 사용자가 이미 제공한 정보가 세션 파일과 작업 문맥에 남아 있다면, 동일 질문을 반복하지 않는다.

## PROGRESS.md

- `./store/<session_id>/PROGRESS.md`
- 언제 무엇이 바뀌었는지를 날짜 순으로 누적 기록하는 작업 로그이다.
- Codex, Claude는 및 사람(리뷰어, 관리자)이 프로젝트 흐름을 빠르게 파악하는 용도이다.
- 규칙
  - 다음 형식을 기본으로 유지한다:
    - Done: 완료된 작업(티켓/PR 링크 포함)
    - In progress: 진행 중인 작업
    - Blocked: 막힌 작업과 원인
    - Next: 다음 세션에서 우선 처리할 작업
  - 중요한 변경, 리스크, 이슈가 발생한 날은 반드시 기록한다.
  - 티켓 ID를 기록할 수 있으면 함께 남긴다.
  - 결정이 수반된 경우, DECISIONS.md의 해당 항목을 링크한다.
  - 최종 결정의 근거는 PROGRESS 대신 DECISIONS.md 파일에 상세히 서술한다.
  - 계획 범위나 완료 기준을 수정하는 대신 일관되게 유지한다.

## DECISIONS.md

- `./store/<session_id>/DECISIONS.md`
- 프로젝트 진행 중 내려진 중요한 결정과 그 이유를 기록한다.
- "왜 이렇게 했는가?"에 대한 단일 참조 지점이다.
- 각 항목에 대해서 상세 정보는 `./store/<session_id>/decisions` 폴더 내 중첩 폴더 없이 파일로 관리한다.
  - 파일 규칙: `DECISIONS-####-{제목}.md`
    - 파일 내 날짜를 분 단위까지 포함하여 명시한다.
- 규칙
  - 다음 중 하나라도 해당되면 즉시 작성 또는 갱신한다.
    - 범위 또는 완료 기준 변경
    - 기술/구현 방식 선택
    - 되돌리기 어려운 결정
    - 팀 내 이견이 있었던 쟁점의 결론
    - 성능/보안/비용/운영에 영향을 주는 선택
  - 결정은 결정 직후 기록하는 것을 원칙으로 한다.
  - 각 결정에는 다음 정보를 포함한다.
    - 결정 일자
    - 문제 상황(컨텍스트)
    - 고려한 대안
    - 최종 결정
    - 결정 근거 및 트레이드오프
    - 예상 영향
  - 관련 티켓, PR, 연구 자료(RESEARCH.md)를 링크한다.
  - 작업 진행 로그는 DECISIONS 대신 PROGRESS.md에 작성한다.
  - 단순 참고 링크의 경우 DECISIONS 대신 RESEARCH.md의 사용법을 확인한다.

## RESEARCH.md

- `./store/<session_id>/RESEARCH.md`
- 작업 및 결정에 실제로 사용한 외부 자료를 프로젝트 맥락에 맞게 정리한다.
- Codex, Claude가 작업 이해도를 높이기 위해 참고해야 하는 문서이다.
- **`./instructions/RESEARCH.md` 파일을 참고하여 가이드 참고**
- 최종 결론이나 결정을 RESEARCH 보다는 DECISIONS.md 파일에서 결정한다.

## 임시 파일

- PC 루트에 접근하는 대신 `./store/<session_id>/temps` 폴더에 기록한다.
  - "/tmp" 위치도 사용하지 않음.
  - Playwright MCP에서 발생하는 파일도 PC 루트 대신 `./store/<session_id>/temps` 위치여야 한다.

- 세션 단위 재사용 캐시와 임시 산출물 전용 폴더이다. 다음을 `temps`에 둘 수 있다.
  - raw API 응답 원본
  - 조사 중간 산출물
  - 일회성 로그, 스크린샷, 비교 결과
  - 재호출 방지용 캐시
- 다음은 `temps`에 두지 않는다.
  - 실제 애플리케이션 소스코드
  - Next.js/백엔드/테스트 프로젝트 스캐폴드
  - 장기 유지되는 fixture source-of-truth
  - Redis/DB 데이터 디렉터리 같은 런타임 자산
- 실제 프로젝트 자산은 항상 레포 루트 또는 소스 디렉터리 아래에 두고, `temps`는 source-of-truth로 취급하지 않는다.
