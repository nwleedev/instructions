# 소스 분석 하네스 ANTI_PATTERNS

이 문서는 전수 분석 작업에서 반복적으로 발생하는 실패 패턴과 더 나은 대안을 함께 정의한다.

## 범위 관리

| 케이스 | Anti | Good |
|---|---|---|
| 일부 파일 샘플링 | 2~3개 파일만 읽고 전체 동작을 추정 | 실제 파일 수를 먼저 확정하고 전부 순회 |
| 무음 제외 | 바이너리, 설정, 테스트 파일을 이유 없이 제외 | 제외가 필요하면 이유와 개수를 ledger에 명시 |
| 비교 대상 누락 | 원본 소스만 보고 공개 배포물 차이를 말함 | 공개 번들/패키지/문서와 대응 관계를 별도 기록 |

## 카탈로그 품질

| 케이스 | Anti | Good |
|---|---|---|
| 파일 설명 생략 | 경로 목록만 남기고 역할을 쓰지 않음 | 각 파일의 역할과 서브시스템을 한 줄 이상 기록 |
| 심볼 누락 | 클래스만 적고 메서드/함수는 생략 | 선언 심볼은 종류별로 모두 기록 |
| 0-symbol 숨김 | 심볼이 없거나 추출 실패한 파일을 건너뜀 | `0 symbols` 또는 추출 한계를 명시 |

## 해석 오류

| 케이스 | Anti | Good |
|---|---|---|
| 문자열 과해석 | 문자열 한 번 발견하고 정책/전략을 단정 | 인접 코드와 호출 맥락까지 확인 후 조건부로 기록 |
| 번들 부재 과해석 | 공개 번들에 없으니 절대 쓰이지 않는다고 결론 | 빌드 제거, 조건 분기, 압축 가능성을 함께 기록 |
| provenance 혼합 | 공개 문서와 비공개 artifact를 같은 강도로 사용 | 증거 강도와 provenance를 따로 표시 |

## 문서 구조

| 케이스 | Anti | Good |
|---|---|---|
| 거대한 단일 문서 | 모든 파일과 심볼을 한 문서에 몰아 넣음 | 개요와 shard 문서를 분리 |
| 요약-원장 혼합 | 사람이 읽는 요약에 exhaustive 목록을 섞음 | 서브시스템 요약과 ledger를 구분 |
| claim 없는 카탈로그 | 카탈로그를 만들었지만 기존 조사와 연결되지 않음 | claim crosswalk를 별도 문서로 유지 |

## 검증

| 케이스 | Anti | Good |
|---|---|---|
| coverage 미검증 | 파일 수가 맞는지 확인하지 않고 종료 | 실제 파일 수와 카탈로그 수를 대조 |
| 표본 재검증 없음 | 자동 추출 결과를 전혀 샘플 확인하지 않음 | 핵심 파일 표본을 사람이 대조 |
| 세션 artifact 누락 | 하네스 생성 후 validation artifact 없이 종료 | 세션 경로에 validation report를 남김 |

## 추가 금지 규칙

- 전수 분석을 했다고 주장하면서 coverage ledger가 없는 상태로 종료하지 않는다.
- “아마”, “추정상” 같은 표현을 사실 카탈로그 항목의 제목으로 쓰지 않는다.
- 내부 전용 신호와 일반 기능 설명을 같은 문단에 섞지 않는다.
