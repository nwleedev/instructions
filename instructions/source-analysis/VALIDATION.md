# 소스 분석 하네스 VALIDATION

## 목표

전수 분석이 실제로 exhaustive했는지, 그리고 생성한 문서가 다음 작업의 기초로 충분한지 확인한다.

## 완료 체크리스트

### 1. 범위와 coverage

- 입력 루트와 비교 루트가 문서에 적혀 있다.
- 실제 파일 수가 계산돼 있다.
- `FILE-INVENTORY` 총합과 실제 파일 수가 일치한다.
- `COVERAGE-LEDGER`에 누락 파일이 `0`으로 표시된다.
- 의도적 제외 항목이 있으면 개수와 이유가 있다. 없으면 `없음`으로 명시한다.

### 2. 파일 카탈로그

- 모든 파일에 역할 요약이 있다.
- 서브시스템 분류가 비어 있는 파일이 없다.
- import/export 요약이 비어 있으면 이유가 있다.
- 신호 태그가 필요한 파일은 직원 전용/내부 인프라/공개 표면 관련성이 기록돼 있다.

### 3. 심볼 카탈로그

- 함수/클래스/메서드가 있는 핵심 파일은 최소 1개 이상의 심볼 엔트리를 가진다.
- 메서드가 많은 클래스는 클래스만 적고 끝내지 않는다.
- 심볼이 없는 파일은 `0 symbols` 또는 추출 한계로 명시한다.
- 샘플 3개 이상을 골라 원본 선언 수와 카탈로그 수를 대조한다.

### 4. 서브시스템 맵

- top-level 주요 디렉터리 책임이 보인다.
- 진입점과 데이터 흐름이 최소 한 번 이상 요약된다.
- 규모가 큰 서브시스템은 별도 shard나 요약으로 분리돼 있다.

### 5. 공개 표면 차이

- 원본 소스와 공개 배포물의 관계가 정리돼 있다.
- 공개 문서에 있는 기능과 내부 artifact에서만 관찰된 신호를 분리한다.
- 고영향 해석은 provenance 수준이 함께 표시된다.

### 6. 세션 지속성

- 결과물이 세션 `temps/` 경로에 정리돼 있다.
- validation artifact가 세션 경로에 존재한다.
- 다음 작업자가 `INDEX -> COVERAGE-LEDGER -> SUBSYSTEM-MAP` 순으로 재개 가능하다.

## 종합 판정 기준

- **통과**
  - coverage 누락 0건
  - 파일/심볼/서브시스템/diff/claim overlay 묶음 존재
  - 표본 검증 완료
- **조건부 통과**
  - coverage는 닫혔지만 일부 심볼 추출 또는 diff 대응이 약한 경우
  - 약한 지점을 명시하고 후속 티켓이 열린 경우만 허용
- **실패**
  - 일부 파일 미분석
  - 카탈로그 수와 실제 파일 수 불일치
  - validation artifact 부재

## 권장 검증 순서

1. 실제 파일 수 계산
2. coverage ledger 대조
3. 표본 파일 직접 점검
4. 공개 번들/문서 대응 점검
5. claim crosswalk와 strict/aggressive 조사 충돌 여부 확인
