# 조사 하네스 ANTI_PATTERNS

## 금지 패턴

### 1. 구현 점프

- 조사 요청을 받았는데 곧바로 UI 설계나 구현으로 넘어간다.
- 아직 범위와 완료 기준이 닫히지 않았는데 코드를 쓰기 시작한다.

왜 문제인가:
- 조사 하네스 설계 목표가 흐려진다.
- 잘못된 전제를 코드로 굳히게 된다.

### 2. 템플릿 감옥

- 템플릿 빈칸을 채우는 것이 목적이 된다.
- 주제에 맞지 않는 섹션까지 억지로 채운다.

왜 문제인가:
- 진짜 중요한 정보가 희석된다.
- 조사 결과가 틀에 맞춘 요약문으로만 남는다.

대응:
- 선택 모듈은 필요할 때만 활성화한다.
- 섹션이 비면 제거한다.

### 3. 단일 검색 경로 의존

- 검색 엔진 한 번 호출한 결과만 믿는다.
- MCP 결과를 그대로 정답처럼 사용한다.

왜 문제인가:
- 검색 랭킹 편향이 그대로 들어온다.
- 최신성, 정확성, 범위 누락을 잡지 못한다.

### 4. 약한 근거의 재포장

- 블로그 글을 사실 확인 없이 근거로 쓴다.
- 벤더 마케팅 문구를 아키텍처 사실처럼 취급한다.

왜 문제인가:
- 결론의 신뢰도가 무너진다.
- 반박 가능성이 높은 문서를 세션 기준으로 굳히게 된다.

### 5. 근거와 결론 혼합

- `이 문서가 이렇게 말한다`와 `그래서 우리에게 맞다`를 같은 문장에 섞는다.
- 가설, 관측, 결론을 구분하지 않는다.

왜 문제인가:
- 후속 검토자가 무엇을 반박해야 하는지 알기 어렵다.

### 6. 운영 이슈의 추측화

- 로그, 재현 조건, 영향 범위 없이 원인부터 단정한다.
- 사용자가 본 증상과 시스템 내부 증상을 구분하지 않는다.

왜 문제인가:
- 잘못된 우선순위와 잘못된 조치로 이어진다.

### 7. 컨텍스트 팽창 방치

- 긴 인용문과 원문 복사를 계속 누적한다.
- 중간 마일스톤 기록 없이 세션에만 의존한다.

왜 문제인가:
- 압축 후 핵심 맥락이 사라진다.
- 같은 조사를 다시 반복하게 된다.

### 8. MCP 종속 설계

- 특정 MCP가 없으면 조사 절차 자체가 돌아가지 않게 만든다.
- Tavily 예시를 곧바로 하네스 표준으로 고정한다.

왜 문제인가:
- 도구가 바뀌면 문서가 바로 무용지물이 된다.

### 9. 조사 범위 무한 확장

- 완료 기준이 없는데 계속 자료를 추가한다.
- 새로 발견한 흥미로운 주제를 원래 목표보다 우선한다.

왜 문제인가:
- 세션이 끝나도 산출물이 닫히지 않는다.

## Anti/Good 필수 쌍 — 출처 관리

### 케이스: 블로그 단독 근거

**Anti (금지)**:
블로그 글 하나로 기술적 결론을 확정한다. 예: "이 블로그에서 X가 가장 빠르다고 했으므로 X를 선택한다."

**Good (권장)**:
공식 문서로 확인한 후, 블로그는 보조 자료로만 사용한다. 예: "공식 벤치마크와 릴리스 노트에서 X의 성능을 확인했고, 블로그 Y는 실사용 경험으로 이를 보강한다."

### 케이스: 마케팅 문구 인용

**Anti (금지)**:
제품 마케팅 문구를 기술적 사실로 취급한다. 예: "앱 스토어 설명에 '혁신적 AI 기반'이라 되어 있으므로 AI를 사용한다고 판단한다."

**Good (권장)**:
마케팅 문구는 사용 방식의 단서로만 쓰고, 기술 결론은 공식 문서, 리뷰, 실제 사용 사례로 기반한다. 예: "앱 설명에 AI를 언급하나, 실제 리뷰와 기술 문서를 통해 적용 범위와 정확도를 확인한다."

### 케이스: 최신성 미확인

**Anti (금지)**:
날짜 확인 없이 자료를 인용한다. 예: "이 보고서에 따르면 시장 규모가 X이다." (보고서 발행일 미확인)

**Good (권장)**:
게시/수정 날짜, 버전, 현재 시점 재검색을 포함한다. 예: "Statista 2025년 3월 보고서 기준 시장 규모는 X이다. data.ai 2024 Q4 데이터와 교차검증했다."

## Anti/Good 필수 쌍 — 조사 폭

### 케이스: 단일 검색 종료

**Anti (금지)**:
첫 검색 결과만으로 결론을 내린다. 예: "앱스토어에서 검색한 상위 3개 앱만 비교하고 분석을 종료한다."

**Good (권장)**:
Exploration Expansion Rule을 적용하여 관점을 확장한다. 예: "앱스토어 상위 앱 외에, 해외 시장 앱, 오픈소스 대안, 커뮤니티 추천 도구도 탐색하고, 사용자 리뷰에서 반복되는 불만 패턴을 별도로 분석한다."

### 케이스: 편향된 결론

**Anti (금지)**:
한 방향 근거만 수집한다. 예: "이 아이디어가 좋다는 근거만 모으고, 실패 가능성이나 기존 솔루션의 강점은 무시한다."

**Good (권장)**:
Contradiction Rule을 적용하여 반대 근거를 필수로 기록한다. 예: "이 아이디어의 가치를 뒷받침하는 근거와 함께, 기존 솔루션이 이미 이 문제를 잘 해결하고 있는 영역, 타겟 사용자의 낮은 전환 의사, 기술적 제약도 분석한다."

### 케이스: 결론-근거 혼합

**Anti (금지)**:
근거 메모와 최종 결론을 같은 문단에 작성한다. 예: "사용자 리뷰에서 불만이 많으므로 이 문제를 해결하면 반드시 성공한다."

**Good (권장)**:
Claim-to-Evidence Map으로 분리 기록한다. 예: "[근거] 앱스토어 리뷰 300건 중 42%가 X 기능 부재를 불만으로 언급. [해석] X 기능에 대한 수요가 있으나, 기존 사용자의 전환 비용과 경쟁 앱의 대응 속도를 별도로 평가해야 한다."

## Anti/Good 필수 쌍 — 산출물

### 케이스: 템플릿 강제 평탄화

**Anti (금지)**:
조사 내용을 템플릿에 맞추기 위해 억지로 변형한다. 예: "비교 축 표에 맞추기 위해 정성적 분석 결과를 점수로 변환한다."

**Good (권장)**:
템플릿 선택 섹션으로 유연하게 구성한다. 예: "정성적 분석이 핵심인 항목은 서술형으로 남기고, 정량 비교가 가능한 항목만 표로 정리한다."

### 케이스: 출처 생략

**Anti (금지)**:
결론만 남기고 출처를 기재하지 않는다. 예: "시장 규모가 크고 성장 중이다."

**Good (권장)**:
최종 문서에 출처와 설계 근거를 직접 포함한다. 예: "시장 규모: Statista (2025.03) 기준 $X.XB, CAGR Y% (data.ai 2024 Annual Report 교차검증)."

## Anti/Good 쌍 — 제품 기획 조사 전용

### 케이스: "멋있어 보이는" 아이디어 함정

**Anti (금지)**:
AI/최신 기술 과시용 기능에 매몰된다. 예: "블록체인으로 투명성을 확보한다", "LLM으로 모든 상담을 자동화한다" — 기술이 실제 사용자 문제를 해결하는지 검증하지 않는다.

**Good (권장)**:
사용자 문제에서 출발하여 기술은 해결 수단으로만 선택한다. 예: "사용자가 겪는 정보 비대칭 문제를 해결하기 위해 공공 데이터를 가공하여 비교 기능을 제공한다. 기술은 REST API + 정적 사이트로 충분하다."

### 케이스: 실사용자 부재

**Anti (금지)**:
실제로 이 문제를 겪는 사람이 충분하지 않은데 아이디어를 밀어붙인다. 예: 커뮤니티 글 2~3개만으로 "많은 사람이 원한다"고 결론 내린다.

**Good (권장)**:
사용자 세그먼트를 구체적으로 정의하고, 정량 근거를 확보한다. 예: "앱스토어 리뷰 분석 + 정부 통계 + 커뮤니티 빈도 분석을 통해 월간 활성 사용자 기반이 최소 N만 명 이상인 문제 영역을 선별한다."

### 케이스: 과도한 범위

**Anti (금지)**:
1인 개발로 실현 불가능한 규모의 아이디어를 선정한다. 예: "종합 플랫폼으로 모든 사회 문제를 한 곳에서 해결한다."

**Good (권장)**:
MVP 범위를 먼저 정의하고, 1인 개발 실현 가능성을 기술 스택 수준에서 검증한다. 예: "핵심 기능 1~2개로 MVP를 정의하고, Next.js + Supabase로 2주 내 프로토타입 가능 여부를 확인한다."

### 케이스: 차별화 착각

**Anti (금지)**:
기존 앱과 실질적 차이가 없는데 다르다고 판단한다. 예: "UI가 더 예쁘면 차별화다", "한국어를 지원하면 차별화다."

**Good (권장)**:
기존 솔루션의 구체적 공백을 식별하고, 해당 공백이 사용자에게 실질적 가치를 주는지 검증한다. 예: "기존 앱 A는 X 기능이 없고, 앱 B는 X 기능이 있으나 정확도가 낮다 (리뷰 근거). 우리는 공공 데이터 연동으로 X 기능의 정확도를 높인다."

### 케이스: 설계 문서 추상화

**Anti (금지)**:
"좋은 UX 제공", "직관적 인터페이스", "사용하기 쉬운 앱" 수준의 모호한 설계로 끝난다.

**Good (권장)**:
Claude Code가 즉시 작업할 수 있는 구체성을 갖춘다. 예: "메인 화면에서 사용자가 우편번호를 입력하면, 반경 5km 내 관련 시설 목록을 지도 위에 표시한다. 각 시설 카드에는 이름, 거리, 운영시간, 사용자 평점이 표시된다. API: GET /facilities?zipcode={zipcode}&radius=5km"

### 케이스: 사회적 이슈 표면화

**Anti (금지)**:
구조적 원인을 파악하지 않고 증상만 다룬다. 예: "고령자가 디지털 서비스를 못 쓰니까 글씨를 크게 만든다."

**Good (권장)**:
구조적 원인을 분석하고, 기술로 개선 가능한 지점을 구체적으로 식별한다. 예: "고령자 디지털 소외의 구조적 원인: (1) 복잡한 인증 절차, (2) 전문 용어 중심 UI, (3) 오류 시 복구 경로 부재. 기술적 개선 지점: 간편 인증 + 단계별 가이드 + 실수 방지 확인 흐름."

## 차단 규칙

- 비교 조사인데 비교 축이 없다면 진행 중지 후 축부터 만든다.
- 운영 이슈 조사인데 증상과 영향 범위가 없다면 원인 결론을 보류한다.
- 출처가 공식 문서 하나도 없다면 1차 결론을 금지한다.
- 세션 재개 정보가 기록되지 않았다면 장시간 조사로 확장하지 않는다.
- 제품 기획 조사에서 기존 솔루션 비교 없이 차별화를 주장하면 진행 중지 후 비교 분석부터 수행한다.
- 사회적 이슈 조사에서 구조적 원인 분석 없이 솔루션을 제안하면 원인 분석부터 수행한다.
- 1인 개발 실현 가능성 검증 없이 설계 문서를 확정하지 않는다.
- 사용자 세그먼트가 정의되지 않은 상태에서 기능 목록을 확정하지 않는다.
- 설계 문서에 모호한 표현("좋은 UX", "직관적", "사용하기 쉬운")만 있으면 구체적 동작 서술로 대체한다.
