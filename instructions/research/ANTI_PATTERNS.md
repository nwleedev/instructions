# `instructions/research/ANTI_PATTERNS.md`

## 목적

이 문서는 조사 작업에서 Codex가 반복적으로 빠지기 쉬운 금지 패턴을 정리한다.

## 금지 1. 조사 요청을 구현 요청으로 해석

이유:

- 사용자가 원하는 것은 설계 이전의 문제 공간 파악일 수 있다.
- 성급한 구현 착수는 조사 범위를 좁히고, 잘못된 전제를 코드로 굳힌다.

대체:

- 먼저 조사 질문, 비교 축, 완료 기준을 정리한다.
- 조사 결과가 설계나 구현으로 이어질 때만 다음 단계로 넘어간다.

## 금지 2. 요약만 남기고 상세 근거를 버리기

이유:

- 텍스트 축약이 발생하면 왜 그런 결론이 나왔는지 추적할 수 없다.
- 다음 세션에서 같은 조사를 다시 반복하게 된다.

대체:

- Summary Notes와 Evidence Notes를 분리한다.
- 링크, 날짜, 관찰 사실, 추론 여부를 남긴다.

## 금지 3. 블로그/큐레이션 글 위주로 결론 내리기

이유:

- 2차·3차 가공 자료는 최신성, 정확성, 예외 조건이 약하다.

대체:

- 공식 문서, 표준 문서, 실제 서비스 화면, 실제 소스 코드를 우선한다.

## 금지 4. 한 가지 조사 방식만 모든 작업에 강제

이유:

- 어떤 조사는 문서가 핵심이고, 어떤 조사는 코드가 핵심이며, 어떤 조사는 런타임 관찰이 핵심이다.
- 한 가지 방식만 고정하면 중요한 증거를 놓치기 쉽다.

대체:

- 조사 시작 시 Research Mode를 고른다.
- 필요한 모드만 조합한다.

## 금지 5. 한 개 사례만 보고 일반화

이유:

- 특정 제품의 제약이나 디자인 취향을 업계 표준으로 오인할 수 있다.

대체:

- 단일 사례면 단일 사례라고 명시한다.
- 일반화가 필요하면 추가 비교 대상을 확보한다.

## 금지 6. 충돌하는 증거를 요약 과정에서 삭제

이유:

- 현실의 서비스는 서로 다른 전략을 취한다.
- 충돌 자체가 설계 선택지를 만드는 핵심 정보일 수 있다.

대체:

- `Tensions / Contradictions` 섹션을 유지한다.
- 무엇이 확정이고 무엇이 미정인지 나눈다.

## 금지 7. 템플릿을 답안처럼 고정

이유:

- 구조는 안정성을 주지만, 과한 구조는 관찰 범위를 좁힌다.
- 새로운 조사 축이 나와도 기록하지 못하게 만든다.

대체:

- 필수 골격만 고정한다.
- 본문 관점 수, 표 형식, 비교 축은 문제마다 조정한다.

## 금지 8. 상세 근거를 한 덩어리 메모에 몰아넣기

이유:

- 비교 대상이 늘어나면 어느 근거가 어느 대상에 속하는지 흐려진다.
- 장문 조사에서는 다시 축약과 맥락 손실이 발생한다.

대체:

- 대상별, 출처별, 모드별로 Evidence Notes를 분리한다.
- `Evidence Index`에 상세 근거 위치를 남긴다.

## 금지 9. 한 관점만 깊게 파고들고 조사 완료 처리

이유:

- 깊이는 있어 보여도, 대안이나 반례를 보지 않으면 결론이 쉽게 편향된다.
- “광범위한 조사”라는 목표를 구조적으로 만족하지 못한다.

대체:

- Coverage Contract를 먼저 세운다.
- 최소 관점 수, 비교 대상 수, 반대 근거 존재 여부를 체크한다.

## 금지 10. 핵심 주장을 근거 없이 서술

이유:

- 문장은 그럴듯해도, 나중에 검토하거나 반박할 수 없는 결론이 된다.
- 세션 재개 시 같은 검증을 반복하게 된다.

대체:

- 핵심 주장마다 Claim ID를 붙인다.
- 각 주장에 연결된 Evidence ID 또는 출처를 남긴다.

## 금지 11. 브라우저로 확인 가능한 것을 추측으로 대체

이유:

- 특정 서비스가 언급됐는데 실제 화면을 보지 않으면 잘못된 전제를 세우기 쉽다.

대체:

- 브라우저 MCP로 첫 진입 화면, 내비게이션, 핵심 CTA, 오류 상태를 확인한다.

## 금지 12. 조사 완료 기준 없이 끝내기

이유:

- “대충 알았다” 수준으로 끝나면 다음 단계에서 다시 불확실성이 터진다.

대체:

- 어떤 질문에 답했는지
- 어떤 근거를 확보했는지
- 무엇이 아직 미정인지
- 지금 당장 설계에 반영 가능한 것은 무엇인지

위 네 가지를 완료 시점에 반드시 남긴다.

## 금지 13. 화면만 보고 네트워크/콘솔 상태를 무시

이유:

- 현대 웹 앱은 DOM만으로는 인증 전이, 권한 실패, 혼합 콘텐츠, 핵심 API 실패를 다 파악할 수 없다.
- 사용자가 실제로 겪는 문제는 UI보다 네트워크와 콘솔에서 먼저 드러날 수 있다.

대체:

- 브라우저 MCP에서 snapshot과 함께 network requests, console messages를 함께 확인한다.
- 조사 메모에 핵심 API 성공/실패와 인증 상태 전이를 남긴다.
