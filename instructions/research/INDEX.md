# `instructions/research/INDEX.md`

## 목적

이 문서는 Codex가 프로젝트 종류와 무관하게 어떤 주제든 조사 작업을 수행할 때 따라야 하는 조사 전용 진입점이다.

조사 작업의 목표는 단순 요약이 아니다.

- 구현으로 넘어가기 전에 문제 공간을 넓게 본다.
- 조사 대상에 맞는 1차 근거를 우선 확인한다.
- 사람이 읽기 쉬운 요약과, 압축되지 않은 상세 근거를 함께 남긴다.

## 언제 사용하는가

- 라이브러리, 프레임워크, 외부 도구를 비교할 때
- 서비스, 제품, 기능, 사용자 흐름을 조사할 때
- 아키텍처, 데이터 흐름, 구현 전략을 검토할 때
- 코드베이스, API 계약, 실제 소스 구조를 조사할 때
- 표준, 정책, 규정, 운영 제약을 조사할 때
- 장애 원인, 성능 병목, 실패 사례를 조사할 때
- 구현 전에 “무엇을 만들어야 하는가”를 명확히 해야 할 때

## 핵심 원칙

- 출처 우선순위는 공식 문서, 표준 문서, 실제 소스 코드, 실제 동작 관찰, 보조 자료 순서다.
- 조사 대상에 맞는 증거 수집 모드를 먼저 고른다.
- 브라우저 MCP는 선택 가능한 조사 모드 중 하나이지, 모든 조사의 기본값은 아니다.
- 조사 결과는 넓이와 깊이를 함께 확보해야 한다.
- 필수 골격은 고정하되, 조사 본문은 문제에 맞게 유연하게 쓴다.
- 요약만 남기지 말고 상세 근거를 별도 섹션에 보존한다.
- 조사 단계에서는 사용자 승인 없이 설계안 확정이나 구현 착수를 하지 않는다.

## 조사 산출물의 최소 골격

아래 골격은 항상 유지한다.

1. Research Brief
   - 조사 질문
   - 사용자 목표
   - 범위 밖 항목
2. Investigation Plan
   - 조사 대상
   - 선택한 조사 모드
   - 우선 확인할 근거
3. Source Map
   - 확인한 출처
   - 출처별 신뢰도
   - 최신성 메모
4. Findings
   - 관점별 핵심 발견
   - 비교 대상 간 공통점과 차이점
5. Tensions / Contradictions
   - 서로 충돌하는 관찰
   - 아직 단정할 수 없는 항목
6. Recommendation
   - 지금 당장 설계에 반영할 점
   - 추가 확인이 필요한 점
7. Coverage Summary
   - 이번 조사에서 실제로 커버한 관점
   - 의도적으로 제외한 관점
8. Claim-to-Evidence Map
   - 핵심 주장
   - 각 주장에 연결된 근거 ID
   - 확신도와 미확정 사유
9. Evidence Index
   - 상세 근거 파일 또는 섹션 위치
   - 조사 모드별 증빙 위치
10. Appendix
   - 상세 메모
   - 모드별 관찰 로그
   - 링크와 날짜

## Coverage Contract

조사가 비사소한 경우 아래 최소 기준을 만족하도록 설계한다.

- 단일 사실 확인이 아닌 조사라면 최소 3개 이상의 관점을 다룬다.
- 비교 조사라면 최소 2개 이상의 대안 또는 사례를 다룬다.
- 추천이나 결론을 제시한다면 최소 1개 이상의 반대 근거, 예외, 또는 미확정 쟁점을 남긴다.
- 구현 전 조사라면 문제 정의, 제약 조건, 대안, 리스크, 검증 방법 중 최소 4개를 다룬다.

이 기준은 최소선이다.
조사 대상이 크거나 중요할수록 더 넓게 커버한다.

## 템플릿 강도 규칙

- 위 골격은 유지한다.
- 하지만 아래 항목은 고정하지 않는다.
  - 관점 개수
  - 비교 대상 개수
  - 장단점 분류 방식
  - 본문 길이
- 조사 주제가 커질수록 본문은 자유롭게 확장하되, 핵심 요약과 근거 추적성은 유지한다.
- 조사 규모가 커지면 근거를 출처별 또는 대상별로 분리 저장할 수 있다.
- Coverage Summary와 Claim-to-Evidence Map은 조사 폭과 구체성을 유지하기 위한 필수 장치로 유지한다.

## 프론트엔드 조사와의 관계

- 구현 규칙은 `instructions/frontend/*`가 담당한다.
- 조사 규칙은 `instructions/research/*`가 담당한다.
- “구현 전에 무엇을 조사해야 하는가”는 research 문서를 따르고, “어떻게 구현해야 하는가”는 frontend 문서를 따른다.
