# `instructions/research/ARCHITECTURE.md`

## 목적

이 문서는 Codex가 조사 작업을 어떤 순서로 수행하고, 어떤 형식으로 근거를 쌓고, 언제 구현으로 넘어갈 수 있는지를 정의한다.

## 1. 조사 시작 조건

조사 착수 전 최소한 아래를 파악한다.

- 조사 질문
- 사용자 목적
- 결과물이 설계 참고용인지, 의사결정용인지, 구현 착수 전 검토용인지
- 조사 대상의 유형
- 반드시 봐야 할 서비스, 문서, 코드, 표준, 데이터가 있는지

이 항목이 비어 있으면 사용자에게 선택지와 함께 보완 질문을 제시한다.

## 2. 조사 흐름

### 2.1 Framing

- 문제를 한 문장으로 다시 쓴다.
- 어떤 유형의 조사인지 먼저 분류한다.
- 무엇을 비교할지 정한다.
- 무엇을 아직 결정하지 말아야 하는지도 적는다.
- 넓이를 확보하기 위해 어떤 관점을 최소 커버할지 먼저 적는다.

### 2.2 Research Mode Selection

조사 대상에 따라 하나 이상의 모드를 고른다.

- Document / Spec mode
  - 공식 문서, 표준, 논문, 정책, 규정, API 명세를 읽는 조사
- Product / Service mode
  - 서비스 UI, UX, 정보구조, 기능 흐름, 제품 전략을 보는 조사
- Code / Source mode
  - 실제 소스 코드, 저장소 구조, 구현 패턴, 공개 API를 보는 조사
- Tool / Dependency mode
  - 라이브러리, 프레임워크, 외부 도구의 선택과 도입 적합성을 보는 조사
- Runtime / Operational mode
  - 네트워크, 로그, 오류, 성능, 인증 전이, 런타임 거동을 보는 조사

항상 모든 모드를 쓰지 않는다.

### 2.3 Source Acquisition

- 공식 문서부터 확인한다.
- Document / Spec mode에서는 아래를 우선 기록한다.
  - 문서 제목
  - 발행 주체
  - 버전 또는 날짜
  - 적용 범위
- Product / Service mode에서는 아래를 우선 기록한다.
  - 첫 진입 URL
  - 로그인 필요 여부
  - 주요 내비게이션 구조
  - 주요 작업 진입점
  - 오류 상태나 권한 제약
- Code / Source mode에서는 아래를 우선 기록한다.
  - 저장소 또는 파일 경로
  - 공개 API와 내부 구현 경계
  - 반복 패턴과 예외 구현
- Tool / Dependency mode에서는 아래를 우선 기록한다.
  - 문제를 해결하는 방식
  - 도입 비용과 제약
  - 대안과 차이점
- Runtime / Operational mode에서는 아래를 우선 기록한다.
  - 핵심 요청의 네트워크 상태
  - 콘솔 오류 또는 경고
  - 인증/권한 전이
  - 성능 또는 실패 흔적

### 2.4 Research Matrix

조사 주제에 따라 필요한 축만 고른다.

- 문제 정의와 성공 조건
- 적용 범위와 제외 범위
- 핵심 개념과 용어
- 인터페이스 또는 공개 계약
- 주요 흐름 또는 동작
- 제약 조건과 의존성
- 대안과 트레이드오프
- 리스크와 실패 패턴
- 검증 방법과 남은 미정 사항

모든 축을 기계적으로 채우지 않는다.

단, 아래 규칙은 지킨다.

- 비사소한 조사에서는 최소 3개 이상의 축을 실제로 다룬다.
- 결론을 내리는 조사에서는 최소 1개 이상의 반대 관점 또는 예외 사례를 포함한다.
- 비교를 전제로 하는 조사에서는 최소 2개 이상의 비교 대상을 둔다.

### 2.5 Capture

조사 중에는 요약과 원문성 메모를 분리한다.

- Summary Notes
  - 사람이 빠르게 읽을 수 있는 발견 요약
- Evidence Notes
  - 출처 URL
  - 확인 날짜
  - 관찰한 사실
  - 추론이면 추론이라고 표시

이 분리를 유지해야 텍스트 축약이 일어나도 근거가 사라지지 않는다.

아래 조건 중 하나라도 만족하면 Evidence Notes를 별도 파일로 분리한다.

- 비교 대상이 2개 이상이다.
- 주요 출처가 4개 이상이다.
- 조사 결과가 한 번의 응답으로 유지되기 어려울 정도로 길다.
- 서로 다른 조사 모드가 함께 섞인다.

Evidence Notes는 아래 단위 중 하나로 나눈다.

- 출처별
- 비교 대상별
- 쟁점별
- 조사 모드별

한 파일 안에 여러 단위를 섞지 않는 것을 기본값으로 한다.

각 Evidence Note에는 최소한 아래를 남긴다.

- Evidence ID
- 대상 또는 쟁점 이름
- 출처
- 확인 시각 또는 버전
- 관찰 사실
- 추론이면 추론 표시
- 관련 핵심 주장 ID

### 2.6 Coverage Check

Synthesis 전에 Coverage Contract를 실제로 만족했는지 확인한다.

- 지금까지 확보한 관점 수
- 비교 대상 수
- 반대 근거 또는 예외 사례 존재 여부
- 아직 비어 있는 관점

넓이가 부족하면 이 단계에서 추가 조사를 수행한다.

### 2.7 Synthesis

- 관점별 핵심 발견을 묶는다.
- 공통 패턴과 예외 패턴을 구분한다.
- 서로 모순되는 발견은 덮지 않고 그대로 표기한다.
- 지금 바로 설계에 반영 가능한 항목과, 추가 확인이 필요한 항목을 나눈다.

핵심 주장에는 Claim ID를 붙인다.

- C1, C2, C3 ...

각 핵심 주장은 아래 중 하나 이상과 연결돼야 한다.

- Evidence ID
- 공식 문서 섹션
- 실제 코드 경로
- 런타임 관찰 로그

Recommendation의 각 항목도 어떤 Claim ID에 기대는지 표시한다.

### 2.8 Handoff Gate

아래 조건을 만족하기 전에는 구현 단계로 넘어가지 않는다.

- 최소 2개 이상의 신뢰 가능한 근거가 있거나, 단일 1차 출처가 충분히 강하다.
- 선택한 조사 모드별 필수 증빙이 반영돼 있다.
- Coverage Contract를 만족한다.
- 핵심 주장과 근거의 연결이 남아 있다.
- 조사 결과가 권장 방향과 리스크를 함께 담고 있다.
- 남은 미정 사항이 실제 구현 방향을 뒤집을 수준인지 평가돼 있다.

## 3. 권장 패턴

- 필수 헤더는 고정하고, 분석 본문은 자유롭게 쓴다.
- “무엇을 안다 / 무엇을 모른다 / 무엇을 추천한다”를 분리한다.
- 선택한 조사 모드에 맞는 증거를 먼저 모으고, 다른 모드는 필요할 때만 추가한다.
- 브라우저 조사에서는 정보구조와 주요 CTA를 먼저 잡고 세부 상호작용은 그 다음에 본다.
- 요약은 앞에 두고, 상세 근거는 뒤에 둔다.
- 조사 결과는 비교표 하나로 끝내지 말고 서술형 해석을 함께 남긴다.
- 긴 조사에서는 근거를 대상별, 출처별, 모드별로 분리 저장한다.
- 핵심 주장은 Evidence ID 또는 명시적 출처와 연결한다.

## 4. 텍스트 축약 방지 규칙

- 최종 응답용 요약과 세션 보존용 상세 메모를 분리한다.
- 대상별 메모를 섞지 않는다.
- 출처 없는 결론을 만들지 않는다.
- “같아 보인다” 같은 표현은 Evidence Notes에서 근거를 확인한 뒤에만 쓴다.
- 긴 조사일수록 Appendix를 유지한다.
- 긴 조사일수록 `Evidence Index`에서 상세 근거 파일을 명시한다.
- 긴 조사일수록 Claim-to-Evidence Map을 유지한다.
- 요약에 없는 세부 정보가 Evidence Notes에서 재구성 가능해야 한다.

## 5. 조사 모드별 체크포인트

### Document / Spec mode

- 누가 발행했는가
- 어떤 버전 또는 날짜인가
- 무엇에 적용되는가
- 필수 요구사항과 선택 요구사항이 무엇인가

### Product / Service mode

- 진입 페이지가 무엇인가
- 로그인 전과 로그인 후가 어떻게 다른가
- 사용자가 가장 먼저 수행하는 작업은 무엇인가
- 핵심 인터랙션이 무엇인가
- 실패/권한 부족/빈 상태가 화면에 어떻게 드러나는가

### Code / Source mode

- 공개 API와 내부 구현 경계가 어디인가
- 반복되는 구현 패턴이 있는가
- 예외 구현이나 우회 구현이 있는가
- 설계 의도를 추정할 수 있는 주석/테스트가 있는가

### Tool / Dependency mode

- 어떤 문제를 해결하는가
- 도입 비용, 학습 비용, 운영 비용은 무엇인가
- 대안 대비 강점과 약점은 무엇인가
- 우리 제약과 충돌하는 부분이 있는가

### Runtime / Operational mode

- 핵심 API 요청과 콘솔 오류가 어떤 상태를 보여주는가
- 인증/권한 전이가 어떻게 일어나는가
- 실패가 재현 가능한가
- 로그, 네트워크, 성능 지표 중 무엇이 직접 원인을 보여주는가

## 6. 템플릿 남용 방지 규칙

- 조사 질문마다 동일한 비교표를 강제하지 않는다.
- 예시 세션은 시작점일 뿐 정답 템플릿이 아니다.
- 필수 골격은 근거 추적성과 완료 판정을 위한 최소 장치로만 사용한다.
- 조사 가치가 높은 새 관점이 나오면 문서 구조를 확장할 수 있다.
