# 조사 하네스 ARCHITECTURE

## 목표

이 하네스는 `질문 입력 -> 자료 수집 -> 근거 정리 -> 교차검증 -> 세션 보존 -> 후속 실행` 흐름을 안정적으로 반복하기 위한 구조다.

## 구성 요소

### 1. Intake Layer

- 조사 목적을 `무엇을 알고 싶은가`와 `왜 지금 필요한가`로 나눈다.
- 요청을 아래 모드 중 하나 이상으로 분류한다.
  - 비교 조사
  - 운영 이슈 조사
  - 기술 선택 조사
  - 제품/워크플로우 대체 가능성 평가
  - 문서/코드 사실 확인

### 2. Search Layer

- 공식 문서를 1차 소스로 사용한다.
- 공식 문서가 부족할 때만 표준 문서, 실제 소스 코드, 공개 이슈를 보강한다.
- 최신성이 중요한 주제는 현재 시점 기준으로 재검색한다.
- MCP가 있으면 검색/추출 속도를 높이되, MCP가 없어도 같은 절차로 진행 가능해야 한다.

검색 실행 제약:
- 단일 턴에서 병렬 검색 호출은 최대 3~4건으로 제한한다.
- 5건 이상의 검색이 필요하면 2턴 이상으로 나누어 실행한다.
- 각 검색의 max_results는 3~5건으로 제한한다.
- 검색 결과 중 명백히 무관한 항목(PDF, 코드 파일, vocab 파일 등)은 즉시 제외하고 유관 항목만 인용한다.
- 각 배치 완료 후 핵심 발견을 즉시 세션 파일에 기록한 뒤 다음 배치를 진행한다.

기본 검색 절차:
1. 질문을 3~7개의 하위 질문으로 분해한다.
2. 하위 질문마다 기본 검색어를 만든다.
3. 필요 시 도메인 제한 검색어를 만든다.
4. 검색은 3~4건씩 배치로 실행한다. 한 턴에서 5건 이상 병렬 실행하지 않는다.
5. 최신성 이슈가 있으면 현재 시점 재검색을 한다.
6. 원문 확인이 필요한 결과는 문서/페이지를 직접 연다.
7. 필요한 경우 추출 도구를 사용해 본문을 확인한다.
8. 질문별 최소 증빙 예산이 찼는지 확인한다.

### 3. Evidence Layer

- 조사 중 찾은 자료를 전부 저장하지 않는다.
- 최종 판단에 실제 영향을 준 자료만 `RESEARCH.md`에 남긴다.
- 고영향 주장에는 독립 근거를 분리한다.
- 근거와 해석을 분리한다.
- 핵심 질문별로 어떤 근거를 채웠는지 추적한다.
- 주요 주장별로 반대 근거 또는 제한 사항을 함께 남긴다.

### 4. Session Persistence Layer

- `PLANS.md`: 목표, 범위, 완료 기준
- `TICKETS.md`: 지금 해야 할 일
- `PROGRESS.md`: 시간순 작업 로그
- `DECISIONS.md`: 왜 그렇게 했는지
- `RESEARCH.md`: 실제 사용한 외부 근거
- `temps/`: 긴 초안, 임시 수집물, 샘플 `PLANS.md`

규칙:
- 컨텍스트 압축 전에 핵심 상태를 세션 파일에 남긴다.
- 압축 후 다시 이어갈 때 필요한 최소 정보는 다음 5개다.
  - 현재 목표
  - 이미 확인한 사실
  - 근거 링크
  - 남은 단계
  - 다음 즉시 실행 항목

### 5. Validation Layer

- 문서화만으로 끝내지 않는다.
- 조사 모드별 체크리스트로 완성도를 확인한다.
- 비교 조사와 운영 이슈 조사는 검증 포인트가 다르므로 같은 템플릿을 그대로 강제하지 않는다.
- 질문별 최소 증빙 예산과 반대 근거 규칙을 통과해야 조사 종료로 본다.

## 템플릿 경직성 대응

템플릿은 강제형이 아니라 선택형 모듈로 운영한다.

### 필수 모듈

- 목적
- 완료 기준
- 조사 질문
- 출처 전략
- 핵심 주장과 근거
- 후속 액션

### 선택 모듈

- 비교 축 표
- 사용 맥락 메모
- 의사결정 영향 메모
- 운영 이슈 타임라인
- 재현 조건
- 비용/성능 표
- 보안 및 권한 메모
- MCP 적용 구성

규칙:
- 선택 모듈이 비어 있으면 제거한다.
- 템플릿을 채우기 위해 약한 주장을 만들지 않는다.
- 조사 주제에 맞지 않는 섹션은 남기지 않는다.

## MCP 적용 원칙

- MCP는 조사 속도와 범위를 넓히는 어댑터 계층이다.
- 도구 선택보다 절차 일관성이 우선이다.
- Tavily MCP는 웹 검색과 추출 보강에 적합하지만, 하네스 전체를 Tavily 전용으로 설계하지 않는다.
- MCP 도구 호출은 다음 정보를 기록 가능해야 한다.
  - 어떤 목적에 사용했는지
  - 어떤 입력을 보냈는지
  - 무엇을 얻었는지
  - 얻은 결과를 어떻게 검증했는지

권장 사용 패턴:
- 검색 폭 확장: 일반 검색, 뉴스 검색, 도메인 제한 검색
- 구체성 강화: 검색 결과에서 원문 추출
- 재검증: MCP 결과를 공식 문서 또는 실제 소스 코드와 대조

## 비교 조사 흐름

1. 비교 대상 정의
2. 비교 목적과 의사결정 맥락 정의
3. 공통 비교 축 정의
4. 각 축별 근거 수집
5. 차이점과 공통점 분리
6. 적용 가능성 또는 의사결정 영향 정리

공통 권장 비교 축:
- 문제 적합성
- 실제 사용 맥락
- 제약과 실패 모드
- 적용 가능성 또는 의사결정 영향

소프트웨어/서비스 비교에서만 추가 검토할 축:
- 기능 범위
- 아키텍처 패턴
- 권한/보안 모델
- 운영성과 관측 가능성
- 비용과 복잡도

## 제품/워크플로우 대체 가능성 평가 흐름

1. 사용자가 실제로 겪는 문제를 정리한다.
2. 현재 도구와 대체 후보가 각각 어떤 방식으로 그 문제를 다루는지 정리한다.
3. 세션 연속성, 조사 품질, 장시간 사용성, 통제 가능성 관점에서 비교한다.
4. 완전 대체 가능한지, 부분 대체가 맞는지, 병행 사용이 맞는지 판단한다.
5. 대체 시 필요한 전제 조건과 남는 리스크를 정리한다.

권장 평가 축:
- 긴 세션에서의 맥락 유지
- 세션 재개 가능성
- 조사 과정 기록 가능성
- 출처 기반 검증 가능성
- 인터페이스 마찰과 장시간 사용 피로
- 도구 확장성
- 결과 이식성

## 운영 이슈 조사 흐름

1. 증상 정의
2. 영향 범위와 사용자 체감 정리
3. 재현 조건 수집
4. 관측 데이터와 로그 요구사항 정리
5. 가능한 원인 후보를 근거와 함께 정리
6. 즉시 완화책과 후속 검증 항목 분리

## 컨텍스트 압축 대응 규칙

- 긴 세션에서 툴 호출과 원문 인용이 많아지면 토큰이 빠르게 소모된다.
- 마일스톤 단위로 중간 결과를 문서화한다.
- 원문 발췌 대신 핵심 사실과 링크를 남긴다.
- 다음 세션에서 이어야 하는 질문 목록을 남긴다.
- 세션 안에서만 유지되는 임시 가설은 `temps/`로 분리한다.
- 제품 대체 가능성 평가에서는 `느리다`, `불편하다`, `유지되지 않는다` 같은 체감 문제를 기능 문제와 분리해 기록한다.

## 제품 기획 조사 흐름

실생활 문제 및 사회적 이슈 기반 제품 기획 조사는 다음 흐름을 따른다.

### 1단계: 도메인 탐색 — 문제 후보 발굴

1. 실생활 불편함과 사회적 이슈를 폭넓게 탐색한다.
2. 각 후보에 대해 빈도, 심각도, 영향 범위를 정량/정성으로 초벌 평가한다.
3. 최소 3개 이상의 후보를 근거와 함께 제시한다.
4. 도메인을 사전 지정하지 않은 경우, 다음 소스에서 탐색을 시작한다.
   - 앱스토어/플레이스토어 낮은 평점 리뷰의 반복 불만
   - 정부/국제기구 공식 보고서의 미해결 이슈
   - 커뮤니티(Reddit, 포럼)에서 반복되는 불만 패턴
   - 사회 조사 데이터에서 드러나는 구조적 문제

### 2단계: 사회적 이슈 심층 분석

1. 선정된 후보의 구조적 원인을 파악한다 (증상이 아닌 원인).
2. 현재 대응 현황을 정리한다 (정부 정책, 민간 서비스, NGO 활동 등).
3. 기술로 개선 가능한 지점을 구체적으로 식별한다.
4. 개선 효과의 크기와 지속성을 추정한다.

### 3단계: 사용자 세그먼트 정의

1. 이 문제를 겪는 사용자 그룹을 정의한다.
2. 각 그룹의 특성 (나이, 기술 숙련도, 사용 빈도, 지불 의사)을 파악한다.
3. 1차 타겟 사용자를 선정하고 그 이유를 근거와 함께 기록한다.
4. 타겟 사용자의 현재 대안 행동(workaround)을 파악한다.

### 4단계: 기존 솔루션 비교 분석

1. 각 후보 문제에 대해 기존 앱/서비스를 최소 2개 식별한다.
2. 다음 축으로 비교한다.
   - 핵심 기능과 해결하는 문제 범위
   - 사용자 리뷰 기반 강점/약점
   - 가격 모델과 접근성
   - 기술 스택 (알 수 있는 경우)
   - 사용자 기반 규모와 성장 추세
3. 기존 솔루션의 공백(gap)을 구체적으로 식별한다.

### 5단계: 차별화 전략 수립

1. 기존 솔루션의 공백에서 차별화 포인트를 도출한다.
2. 차별화가 실제 사용자 가치를 제공하는지 검증한다 (기술 과시가 아닌지 확인).
3. 차별화 포인트를 기능 수준으로 구체화한다.
4. 기존 앱과 실질적 차이가 있는지 Contradiction Rule을 적용해 검증한다.

### 6단계: 기술 실현 가능성 검증

1. 1인 개발자가 구현 가능한 범위를 정의한다.
2. 필요한 기술 스택을 식별하고 제약 조건을 정리한다.
3. MVP(최소 기능 제품) 범위를 정의한다.
4. 다음 기준으로 실현 가능성을 판단한다.
   - 핵심 기능의 기술적 난이도
   - 외부 API/서비스 의존도
   - 데이터 소스 접근성
   - 예상 개발 기간 (1인 기준)
   - 유지보수 부담

### 7단계: 설계 문서 산출

1. 최종 선정 아이디어에 대해 Claude Code가 작업 가능한 수준의 설계 문서를 작성한다.
2. 설계 문서에는 다음이 포함된다.
   - 제품 개요와 핵심 가치 제안
   - 기능 목록과 우선순위
   - 기술 스택 선정과 근거
   - 데이터 모델 개요
   - API 설계 개요
   - UI/UX 핵심 흐름
   - MVP 범위와 후속 확장 계획
3. 모호한 표현("좋은 UX 제공", "사용하기 쉬운 인터페이스")을 금지하고, 구체적 동작을 서술한다.
