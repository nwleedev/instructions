# 조사 하네스 VALIDATION

## 기본 원칙

- 완료는 `자료를 많이 모음`이 아니라 `질문에 답할 수 있는 근거 구조가 닫힘`으로 판단한다.
- 결과물은 다음 세션이나 다른 Codex 인스턴스가 이어받아도 이해 가능해야 한다.

## 공통 완료 체크리스트

- 목적과 완료 기준이 문서에 있다.
- 조사 질문이 명시되어 있다.
- 핵심 질문마다 최소 증빙 예산이 채워졌거나, 부족 사유가 기록되어 있다.
- 출처 우선순위가 지켜졌다.
- 핵심 주장마다 근거가 있다.
- 핵심 주장마다 반대 근거, 제한 사항, 불확실성 중 최소 1개가 있다.
- 고영향 주장은 독립 근거가 충분하다.
- 가설, 관측, 결론이 분리되어 있다.
- 다음 액션이 있다.
- 세션 재개에 필요한 정보가 남아 있다.

## 모드별 필수 증빙

### 비교 조사

- 비교 대상 정의
- 비교 축 2개 이상
- 각 축별 근거 링크
- 차이점과 공통점 요약
- 적용 가능성 또는 의사결정 영향 메모

### 운영 이슈 조사

- 증상 정의
- 영향 범위
- 재현 조건 또는 재현 불가 사유
- 가능한 원인 후보와 각 근거
- 즉시 완화책과 추가 확인 항목

### 기술 선택 조사

- 후보군
- 선택 기준
- 장단점
- 도입 리스크
- 우리 프로젝트 적용 판단

### 제품/워크플로우 대체 가능성 평가

- 현재 도구에서 겪는 핵심 문제 정의
- 대체 후보가 그 문제를 어떻게 다루는지에 대한 근거
- 완전 대체 / 부분 대체 / 병행 사용 중 어떤 결론인지
- 대체 시 필요한 전제 조건
- 대체 후에도 남는 리스크와 한계

### 제품 기획 조사 (실생활 문제 및 사회적 이슈 기반)

- 문제 후보 최소 3개 이상 제시, 각 후보에 빈도/심각도/영향범위 근거 포함
- 각 후보에 대해 기존 솔루션 최소 2개 비교 분석 완료
- 사회적 이슈 후보에 대해 구조적 원인 분석 포함 (증상만 나열 금지)
- 사용자 세그먼트가 구체적으로 정의됨 (나이, 기술 숙련도, 사용 빈도, 지불 의사 중 2개 이상)
- 최종 선정 아이디어의 차별화 포인트가 기능 수준으로 구체화됨
- 차별화 포인트에 대해 Contradiction Rule 적용 (기존 솔루션이 이미 해당 가치를 제공하는 경우 기록)
- 1인 개발 실현 가능성이 기술 스택 수준에서 검증됨 (기술 스택, MVP 범위, 예상 기간)
- 설계 문서가 Claude Code의 작업 입력으로 사용 가능한 구체성을 갖춤 (API 경로, 데이터 모델, UI 흐름 중 2개 이상)
- 모호한 표현 없이 구체적 동작이 서술됨

### 수익화 전략 검증 (제품 기획 조사 확장 — 선택 모듈)

수익화가 조사 범위에 포함될 때 아래 항목을 추가로 검증한다.

- 수익 모델 후보가 2개 이상 비교 검토됨
- 경쟁사 가격 데이터가 후보당 최소 2개 수집됨
- 수익 모델 선택 근거가 타겟 사용자 특성(지불 의사, 사용 빈도)과 연결됨
- 단위 경제학 추정이 낙관/기본/비관 3개 시나리오로 제시됨
- 1인 개발자 맥락의 비용 제약(서버, API, 마케팅)이 반영됨
- 수익화 타이밍 결정이 근거와 함께 기록됨
- 선택한 수익 모델에 Contradiction Rule 적용됨 (실패 시나리오 기록)
- 프리미엄 모델 선택 시 전환율 벤치마크가 출처와 함께 제시됨

## 고영향 Claim 검증 강도

아래 항목은 고영향 claim으로 본다.

- 사용자나 팀의 의사결정 방향을 바꾸는 주장
- 현재 도구, 제품, 워크플로우를 대체하거나 병행 사용하도록 유도하는 주장
- 비용, 시간, 리스크, 신뢰도 판단에 영향을 주는 주장
- 운영 대응 또는 우선순위를 바꾸는 주장

소프트웨어/서비스 조사에서는 아래도 고영향 claim으로 추가 본다.

- 아키텍처 방향을 바꾸는 주장
- 새 의존성 도입을 정당화하는 주장
- 성능, 보안, 권한 모델에 영향을 주는 주장

제품 기획 조사에서는 아래도 고영향 claim으로 추가 본다.

- 특정 문제의 시장 규모나 사용자 수를 주장하는 경우
- 기존 솔루션에 특정 공백이 있다고 주장하는 경우
- 차별화 포인트가 사용자에게 실질적 가치를 준다고 주장하는 경우
- 1인 개발로 특정 기간 내 구현 가능하다고 주장하는 경우
- 사회적 이슈의 구조적 원인을 특정하는 경우
- 특정 수익 모델이 타겟 사용자에게 적합하다고 주장하는 경우
- 전환율이나 ARPU를 특정 수치로 추정하는 경우
- 특정 가격대가 경쟁력 있다고 주장하는 경우

검증 규칙:
- 가능하면 독립 근거 2개 이상
- 한 근거가 벤더 문서라면 다른 근거는 표준 문서, 실제 코드, 공개 이슈 중 하나로 보강
- 독립 근거가 하나뿐이면 `확정` 대신 `현재 가장 유력한 해석`으로 표시

## Evidence Notes 분리 규칙

- `Evidence Notes`:
  - 사실, 발췌 요약, 링크, 관찰 결과
- `Working Interpretation`:
  - 현재 해석, 의미, 적용 가능성
- `Decision Candidate`:
  - 아직 결정 전인 선택지와 트레이드오프

규칙:
- `RESEARCH.md`에는 Evidence Notes 중심으로 남긴다.
- 최종 선택은 `DECISIONS.md`에서 확정한다.

## 질문별 최소 증빙 규칙

핵심 질문마다 아래를 확인한다.

- 1차 근거 1개 이상
- 보강 근거 1개 이상
- 최신성 확인 정보 1개 이상

판정:
- 모두 충족: `충분`
- 일부 부족하지만 방향성은 있음: `조건부`
- 핵심 판단에 필요한 근거가 비어 있음: `추가 조사 필요`

규칙:
- `조건부`와 `추가 조사 필요` 질문은 결과 요약에서 따로 표시한다.
- 질문 하나가 비어 있는데 전체 결론을 `확정`으로 쓰지 않는다.

## 반대 근거 규칙

- 주요 주장마다 아래 중 하나 이상을 포함한다.
  - 직접적인 반대 근거
  - 적용 한계
  - 문서 충돌
  - 최신성 리스크
  - 데이터 부족

- 반대 근거가 없는 주장에는 `아직 발견한 반대 근거 없음`과 검색 범위를 기록한다.

## 세션 재개 검증

다음 질문에 답할 수 있으면 재개 가능 상태다.

1. 이 세션의 목표는 무엇인가
2. 지금까지 확인된 사실은 무엇인가
3. 어떤 근거를 사용했는가
4. 아직 남은 질문은 무엇인가
5. 다음 즉시 실행 단계는 무엇인가

## MCP 사용 검증

- MCP가 없어도 같은 조사 절차를 설명할 수 있다.
- MCP를 사용했다면 어떤 입력과 결과가 나왔는지 요약할 수 있다.
- MCP 결과를 다른 근거로 교차검증했는지 설명할 수 있다.
- 민감하거나 파괴적인 작업이 있다면 사용자 승인 규칙이 정의되어 있다.

## 드라이런 기준

샘플 주제 드라이런에서는 아래를 확인한다.

- 템플릿이 결과를 억지로 자르지 않는가
- 비교 조사와 운영 이슈 조사에 같은 뼈대를 적용할 수 있는가
- 결과물을 다른 세션의 `PLANS.md`로 옮길 수 있는가
- 세션 압축 전후로 맥락 손실 없이 이어갈 수 있는가
- 소프트웨어 중심 관점이 아닌 제품/워크플로우 평가에도 같은 하네스를 적용할 수 있는가
- 제품 기획 조사에서 3개 이상의 문제 후보가 근거와 함께 제시되는가
- 기존 솔루션 비교 분석이 실제 차별화 포인트 도출로 이어지는가
- 설계 문서가 모호한 표현 없이 Claude Code가 즉시 작업 가능한 수준인가
- 1인 개발 실현 가능성 검증이 기술 스택/MVP 범위/예상 기간을 포함하는가
- 사회적 이슈의 구조적 원인 분석이 증상-원인-개선지점 구조로 되어 있는가

## 검증 근거

- OpenAI Responses API는 상태 유지형 상호작용과 도구 사용을 전제로 한다.
  - https://developers.openai.com/api/reference/responses/overview
- OpenAI Codex App Server는 `thread/compact/start`와 `contextCompaction`을 제공한다.
  - https://developers.openai.com/codex/app-server/#trigger-thread-compaction
- MCP tools 문서는 사용자 확인, 입력 표시, 결과 검증, 감사 로그를 권장한다.
  - https://modelcontextprotocol.io/docs/concepts/tools
- Nielsen Norman Group의 UX 리서치 방법론은 사용자 세그먼트 정의와 문제 발견 절차를 체계화한다.
  - https://www.nngroup.com/articles/which-ux-research-methods/
- IDEO의 Human-Centered Design 방법론은 사회적 이슈 해결을 위한 디자인 씽킹 프레임워크를 제공한다.
  - https://www.designkit.org/methods
- Statista와 data.ai는 시장 규모, 앱 다운로드, 사용자 행동 데이터의 신뢰할 수 있는 소스이다.
  - https://www.statista.com/
  - https://www.data.ai/
